## 项目实践过程遇到的问题以及解决问题时的思考过程和方法

### 问题一：需求范围与项目案例的平衡

**问题描述**：项目案例共列出 16 项功能模块，但团队只有 3 人、30 天工期，如何在有限资源下合理划定 MVP 范围？

**解决过程**：
1. 首先将所有功能按"核心-增强-锦上"三级分类，识别出学生端求职辅导闭环是企业端、管理端等功能的基石
2. 与需求规格说明书对照，将 R-001~R-014 定为 MVP 必做项，R-015~R-018 及高校后台、完整招聘流程等列为 V2.0 迭代
3. 在概要设计说明书中新增"后续迭代清单"章节，明确标注超出 V1.0 的功能，避免评审时产生"案例功能未体现"的误解

**收获**：学会了在资源受限时做减法，优先保证核心闭环可演示，同时为后续迭代预留扩展点（字段、接口、菜单入口）。

### 问题二：学习进度数据如何存储与聚合

**问题描述**：需求要求展示学生学习进度和成长统计，但进度数据分散在学习任务、测评结果、面试记录、简历分析等多个表中，如何设计高效的聚合方案？

**解决过程**：
1. 最初考虑新增独立的 `learning_progress` 表存储聚合后的进度快照，但会增加数据一致性的维护成本
2. 转而采用"实时聚合"方案：ProgressService 在查询时从 `learning_task`、`assessment_result`、`interview_record`、`resume_analysis` 四张表实时计算进度指标
3. 成长报告采用异步生成 + 缓存策略：AI 生成后存入 `recommendation_record` 表（type=GROWTH_REPORT），后续查询直接从缓存读取

**收获**：理解了"聚合根"设计模式在报表场景的应用，以及实时计算 vs 预计算快照的 trade-off 考量。

### 问题三：AI 服务不可用时的兜底方案

**问题描述**：平台核心功能（职业探索、简历优化、模拟面试、企业推荐、智能客服）都依赖 DeepSeek API，如果 API 不可用，如何保证评审和演示不受影响？

**解决过程**：
1. 为每个 AI 功能设计独立的降级方案：职业探索用规则推荐（测评分数+技能标签），简历优化返回预置检查清单，面试使用本地题库，企业推荐用关键词匹配，客服用 FAQ 匹配
2. 设计统一的 AI 调用链路：Redis 缓存检查 → DeepSeek API 调用 → JSON 解析 → 重试 1 次 → 降级兜底
3. 新增 `ai_call_log` 表记录每次调用的状态、耗时和降级原因，确保 AI 服务可追踪

**收获**：学会了高可用系统的降级设计思路——"依赖外部服务时，永远要有 Plan B"。

### 问题四：Git 中文文件名与路径变更问题

**问题描述**：文档目录从 `系统设计/` 重命名为 `系统架构/` 后，Git 无法正确追踪变更，提交时只显示删除旧文件、不显示新增文件。

**解决过程**：
1. 配置 `git config --global core.quotepath false` 解决中文文件名显示乱码
2. 使用 `git rm --cached "系统设计"` 移除旧路径追踪，再 `git add "系统架构"` 添加新路径
3. 验证提交后远程仓库文件内容正确

**收获**：加深了对 Git 文件追踪机制的理解，以及处理中文路径问题的经验。
