## 项目实践过程遇到的问题以及解决问题时的思考过程和方法

### 问题1：需求评审中发现"学习进度展示"功能缺失
**现象**：评审时指出学生端缺少学习进度闭环，无法展示学习成果
**思考过程**：
- 是否需要新增独立的数据表存储进度？
- 进度数据是否可以由现有表聚合生成？
**解决方法**：
- 采用数据聚合策略，避免新增复杂学习事件表
- 进度由learning_task + assessment_result + interview_record + resume_analysis聚合生成
- 新增3个进度查询接口（GET /api/student/progress/overview/skills/report）
- 成长报告复用recommendation_record表（type=GROWTH_REPORT）
**收获**：数据聚合比新增表更轻量，降低系统复杂度，同时满足功能需求

### 问题2：AI调用缺乏审计机制
**现象**：评审时指出AI服务不可追踪，无法排查问题和优化成本
**思考过程**：
- 需要记录哪些信息？（调用场景、Prompt、响应、耗时、状态）
- 如何平衡隐私保护和可追溯性？（Prompt存摘要而非全文）
**解决方法**：
- 新增ai_call_log表，记录：user_id、scene、prompt_summary、response_source、status、duration_ms、fallback_reason
- 明确AI调用链路：Prompt模板 → 参数组装 → API调用 → JSON解析 → 缓存 → 降级 → 记录日志
- 所有降级方案均记录状态（status=FALLBACK，fallback_reason注明原因）
**收获**：审计日志设计要兼顾隐私和可追溯，摘要+状态+耗时是关键字段

### 问题3：项目案例中功能超出V1.0范围
**现象**：项目案例包含16项功能，但需求规格说明书V1.0仅定义14项
**思考过程**：
- 是否需要在当前版本中实现所有案例功能？
- 如何避免"案例功能未体现"的评审风险？
**解决方法**：
- 明确本次完善只按V1.0实现范围处理，不把案例16项全部纳入
- 新增第6章「后续迭代清单」，将R-015~R-018及高校后台、完整企业招聘流程、语音客服、外部招聘数据接入等明确列为V2.0
- 当前设计预留最小扩展点（字段或菜单入口），确保V2.0无需重大重构
**收获**：架构设计要区分当前交付和后续迭代，预留扩展点比过度设计更重要

### 问题4：Git提交时中文文件名显示异常
**现象**：git status显示中文文件名为转义编码（\347\263\273\347\273\237）
**思考过程**：
- Git默认对非ASCII字符进行转义显示
- 需要配置Git正确显示中文文件名
**解决方法**：
- 执行git config --global core.quotepath false
- 重新添加文件后正常显示中文文件名
**收获**：跨平台开发要注意编码配置，Git的quotepath设置影响中文文件名显示
